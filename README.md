# Foundry NFT Collection

A compact, security-minded NFT project built with Foundry to demonstrate ERC721 development, deployment scripting, dynamic metadata, and testing discipline.

This repository contains two Solidity NFTs:

- `BasicNft`: a minimal ERC721 that mints tokens with IPFS-hosted metadata.
- `MoodNft`: a dynamic ERC721 with fully on-chain metadata and SVG images that change when the token mood flips.

This repo is intentionally small and readable. It is meant to show strong smart contract fundamentals, clean project structure, and the kind of testing and access-control thinking that matters for smart contract development, audits, internships, and junior Solidity roles.

## Contracts

### `BasicNft`

`BasicNft` is a lightweight ERC721 implementation for minting NFTs whose metadata URI is stored per token.

What it demonstrates:

- ERC721 inheritance and mint flow
- Per-token URI storage
- Simple deployment and minting
- IPFS-based metadata usage

Main behavior:

- Mints sequential token IDs starting from `0`
- Stores a token URI for each minted NFT
- Returns the stored URI through `tokenURI`

### `MoodNft`

`MoodNft` is the more interesting contract in the repo. It stores two SVG image URIs and switches between them based on token state.

What it demonstrates:

- On-chain metadata using Base64-encoded JSON
- On-chain SVG image handling
- Stateful NFTs
- Authorization-aware state changes

Main behavior:

- Mints a token with default `HAPPY` mood
- Builds metadata entirely on-chain
- Lets the owner or approved operator flip the token mood
- Updates the NFT image shown in metadata depending on current mood

## Security And Audit Notes

This project is a learning and portfolio project, not a production-audited system. Still, it was built with an audit mindset:

- Uses OpenZeppelin contracts instead of custom ERC721 logic.
- `MoodNft.flipMood` relies on `_checkAuthorized(...)` so only the owner or an approved operator can change state.
- `MoodNft.tokenURI` checks token existence through `ownerOf(tokenId)`.
- SVG file reads are explicitly permitted in `foundry.toml`.
- Core minting and metadata flows are covered by tests.

If this were being prepared for production, the next steps would be:

- Add more negative-path tests for unauthorized actions and invalid token IDs
- Add fuzz tests and invariant-style checks
- Add CI for formatting, build, and test execution
- Add NatSpec documentation across contracts and scripts
- Review event coverage and operational assumptions before deployment

## Getting Started

### Prerequisites

- [Foundry](https://book.getfoundry.sh/getting-started/installation)
- Git

### Clone And Install

If you clone the repository with submodules:

```bash
git clone <your-repo-url>
cd foundry-nft
git submodule update --init --recursive
forge build
```

If you want to install dependencies with Foundry instead:

```bash
forge install Cyfrin/foundry-devops@0.4.0 --no-commit
forge install foundry-rs/forge-std@v1.15.0 --no-commit
forge install OpenZeppelin/openzeppelin-contracts@v5.6.1 --no-commit
forge build
```

## Environment Variables

## Common Commands

Build and test:

```bash
forge build
make test
forge fmt
anvil
```

Deploy and interact locally:

```bash
make deploy
make mint
make deployMood
make mintMoodNft
make flipMoodNft
```

Deploy to Sepolia:

```bash
make deploy ARGS="--network sepolia"
make deployMood ARGS="--network sepolia"
```

Notes:

- `make mint` uses the most recent `BasicNft` deployment for the current chain.
- `make mintMoodNft` and `make flipMoodNft` use the most recent `MoodNft` deployment for the current chain.
- The interaction scripts depend on Foundry broadcast artifacts generated by prior deployments.
